1. Загальна інформація про проєкт

Назва проєкту: CLI Item Catalog
Тип застосунку: консольний (.NET)
Мова програмування: C#
Платформа: .NET
Тип інтерфейсу: Command Line Interface (CLI)
Проєкт є навчальним консольним застосунком, розробленим з метою демонстрації принципів об’єктно-орієнтованого програмування, розділення відповідальностей між компонентами та побудови модульної архітектури програмного забезпечення.

2. Архітектура проєкту

Проєкт побудований з дотриманням принципу Separation of Concerns і логічно поділений на такі шари:
Шар представлення (CLI)
Шар бізнес-логіки
Підсистема збереження (постійність)
Моделі даних
Кожен шар виконує чітко визначену роль та не має жорстких залежностей від інших компонентів.

3. Командний інтерфейс (CLI)
3.1 Призначення CLI

CLI забезпечує взаємодію користувача із застосунком у текстовому режимі.
Він відповідає за:
прийом команд користувача;
валідацію вводу;
виклик відповідних бізнес-операцій;
вивід результатів у консоль.
CLI не містить бізнес-логіки та не зберігає дані.

3.2 Доступні команди
Команда	Опис
help	Виводить список доступних команд
list	Виводить список усіх об’єктів каталогу
create	Створює новий об’єкт через діалог
exit	Завершує роботу застосунку

3.3 Режим роботи
Застосунок працює у режимі REPL (Read–Eval–Print Loop):
Очікує введення команди
Аналізує введення
Виконує команду
Виводить результат
Повертається до очікування наступної команди

4. Бізнес-логіка
   
4.1 Загальна характеристика

Бізнес-логіка відповідає за керування колекцією об’єктів типу Item.
Вона реалізує основні операції над предметною областю та не залежить від способу представлення даних користувачу.
Бізнес-логіка:
ізольована від CLI;
не виконує форматування виводу;
не працює з файловою системою напряму.

4.2 Предметна область

Предметною областю застосунку є каталог об’єктів.
Кожен об’єкт представляє логічну сутність з мінімальним набором властивостей.

4.3 Модель даних

Основною бізнес-моделлю є клас Item.

Властивості:

Id — унікальний ідентифікатор об’єкта

Name — назва об’єкта

Колекція об’єктів зберігається в оперативній пам’яті застосунку.

4.4 Основні бізнес-операції
4.4.1 Створення об’єкта

Отримання назви об’єкта

Валідація введених даних

Створення екземпляра Item

Додавання до колекції

4.4.2 Перегляд каталогу

Повернення поточного списку об’єктів

Без форматування або виводу

4.5 Валідація даних

У межах бізнес-логіки виконується:

перевірка на порожні значення;

контроль коректності створення об’єктів.

4.6 Взаємодія з іншими компонентами

Бізнес-логіка:

не залежить від CLI;

не залежить від підсистеми збереження;

може бути повторно використана з іншим інтерфейсом (GUI, API).

5. Підсистема збереження (Постійність)
5.1 Призначення

Підсистема постійності забезпечує збереження стану каталогу між запусками застосунку.

5.2 Обрана стратегія

Для збереження використовується серіалізація у JSON-файл.

Переваги:

простота реалізації;

відсутність зовнішніх залежностей;

зручність розширення.

5.3 Дані, що зберігаються

список об’єктів Item;

їх ідентифікатори;

назви об’єктів.

5.4 Процес збереження (Save)

Отримання колекції з бізнес-логіки

Серіалізація у JSON

Запис у файл

5.5 Процес відновлення (Load)

Перевірка наявності файлу

Зчитування JSON

Десеріалізація у Item

Відновлення колекції

5.6 Місце зберігання файлів

Файл збереження розміщується у:

Windows: AppData/Local

Linux/macOS: прихована директорія користувача

6. Графічний інтерфейс (проектно)

На даному етапі графічний інтерфейс не реалізований, але передбачений архітектурно.

6.1 Обрана технологія

6.2 Архітектурний підхід

MVVM (Model–View–ViewModel)
